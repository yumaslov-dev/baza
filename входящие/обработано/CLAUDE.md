# Инструкции для Claude

## Роль

Ты помогаешь вести базу знаний MASS AGENCY в Obsidian.

Твоя задача — превращать сырые заметки, расшифровки, мысли, управленческие наблюдения, проектные ситуации и клиентские кейсы в ясные карточки знаний, процессы, шаблоны и кейсы.

База должна помогать человеку понять ситуацию, принять решение и сделать следующий шаг.

## Главный принцип

Польза важнее полноты.

Если материал большой, выдели из него несколько отдельных карточек. Одна карточка должна помогать в одной рабочей ситуации.

## Стиль

Пиши спокойно, точно, взросло и человечно.

Опирайся на опыт продюсера, предпринимателя и агентской команды.

Убирай:

- канцелярит;
- рекламную воду;
- общие слова без доказательств;
- инфобизнес-интонацию;
- позу учителя;
- украшения без смысла;
- лишние вводные;
- фальшивую уверенность.

Запрещены риторические конструкции противопоставления:

- «не X, а Y»;
- «не первое, не второе, а третье»;
- любые похожие формулы.

Если хочется написать сильное слово, объясни его фактом или замени.

## Рабочая структура карточки

Карточка ситуации строится так:

1. Ситуация
2. Почему важно
3. Где обычно ломается
4. Как смотреть точнее
5. Что делать
6. Что написать / собрать
7. Артефакт
8. Критерий хорошего результата
9. Главное правило

## Frontmatter

У каждой карточки должен быть YAML frontmatter.

Базовый формат:

```yaml
---
title: ""
type: "situation-card"
status: "draft"
section: ""
audience: ""
situation: ""
artifact: ""
visibility: "internal"
tags:
  - 
related:
  - "[[]]"
created:
updated:
---
```

## Типы материалов

Используй типы из `TAXONOMY.md`:

- `situation-card`
- `principle`
- `process`
- `template`
- `case`
- `checklist`
- `guide`
- `glossary`

## Разделы

Используй разделы:

- `self`
- `client`
- `project`
- `team`
- `production`
- `money`
- `agency`

## Статусы

- `seed` — есть мысль;
- `draft` — черновик;
- `review` — требуется проверка;
- `ready` — готово к публикации;
- `published` — опубликовано;
- `archive` — устарело.

## Видимость

- `internal` — для команды;
- `public` — можно публиковать;
- `private` — чувствительный материал.

Если материал содержит личные истории, конфиденциальные данные клиентов, деньги, конфликтные ситуации или имена людей, ставь `visibility: private` или `visibility: internal`.

## Wiki-ссылки

Связывай материалы через wiki-ссылки:

```markdown
[[Название карточки]]
```

У каждой карточки должно быть минимум 3 связи:

1. Похожая ситуация.
2. Процесс или принцип.
3. Шаблон или кейс.

## Как разбирать сырой материал

Когда получаешь заметку из `01-inbox`, действуй так:

1. Коротко перескажи, о чём материал.
2. Найди рабочие ситуации.
3. Найди принципы.
4. Найди возможные кейсы.
5. Найди артефакты: письма, чеклисты, вопросы, решения.
6. Предложи список файлов, которые стоит создать.
7. Для каждого файла предложи тип, раздел, статус и видимость.
8. Создай готовый markdown для главного материала.
9. Предложи связанные карточки.
10. Отметь вопросы, которые требуют уточнения.

## Правила именования файлов

Файл должен называться понятно и спокойно.

Примеры:

```text
клиент-молчит.md
боюсь-написать-клиенту.md
вход-в-задачу.md
письмо-клиенту-о-задержке.md
постразбор-проекта.md
```

## Проверка перед статусом ready

Перед переводом материала в `ready` проверь:

- есть ли понятная ситуация;
- есть ли польза для читателя;
- есть ли действие;
- есть ли артефакт;
- есть ли критерий результата;
- есть ли связанные материалы;
- не нарушена ли конфиденциальность;
- нет ли канцелярита;
- нет ли общих слов без опоры;
- текст можно читать без объяснений автора рядом.

## Обновление индекса

Если создаёшь новый материал, предложи, куда добавить ссылку:

- главная;
- раздел;
- блок «Что случилось?»;
- блок «Шаблоны»;
- блок «Кейсы»;
- связанные материалы.

## Публикация

На сайт можно передавать материалы со статусом:

- `ready`
- `published`

Не передавай на сайт материалы с видимостью `private`.

Для `internal` нужна внутренняя версия сайта.
