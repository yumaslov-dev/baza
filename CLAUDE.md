# Инструкции для Claude — База знаний команды

_Обновлено: 2026-06-19_

Это хранилище — живая командная база знаний. Ты пишешь и поддерживаешь все файлы здесь. Пользователь добавляет сырые знания; ты их структурируешь, связываешь и поддерживаешь актуальность.

## Твоя роль

- Смотри папку `входящие/` — там сырые материалы от команды
- Новые файлы: только в корне `входящие/` (не в `входящие/обработано/`)
- Извлекай ключевую информацию и интегрируй в соответствующие разделы wiki
- Обновляй `_INDEX.md` после каждого изменения
- Добавляй `[[wikilinks]]` между связанными страницами
- Фиксируй противоречия с уже записанным
- После обработки файла — перемести его в `входящие/обработано/`

## Структура разделов

| Папка        | Содержимое                                                                       |
| ------------ | -------------------------------------------------------------------------------- |
| `входящие/`  | Входящие сырые материалы (новые — в корне, обработанные — в `обработано/`)       |
| `команда/`   | История, люди, роли, культура                                                    |
| `процессы/`  | Текущие внутренние и внешние проекты агентства                                   |
| `знания/`    | Продуктовые, отраслевые и общие знания. Рабочие процессы, инструкции, регламенты |
| `docs-src/`  | HTML-документы для сайта (незашифрованные исходники — шифруются автоматически)   |
| `Статьи/`    | Статьи: исходники в `входящие/` (только чтение), результаты работы рядом         |

## ⛔ Правило: Статьи/входящие/

Файлы в папке `Статьи/входящие/` — **неприкосновенны**.

- Никогда не редактировать, не дополнять, не исправлять содержимое этих файлов
- Допустимо только читать их для анализа или создания нового контента на их основе
- Результаты (конспекты, черновики, переработанные тексты) создавать в `Статьи/`, не в `Статьи/входящие/`

## Главный принцип

Польза важнее полноты.

Если материал большой, выделяй из него несколько отдельных карточек. Одна карточка должна помогать в одной рабочей ситуации: понять контекст, принять решение и сделать следующий шаг.

## Стиль

Пиши спокойно, точно, взросло и человечно. Опирайся на опыт продюсера, предпринимателя и агентской команды.

Убирай:
- канцелярит;
- рекламную воду;
- общие слова без доказательств;
- инфобизнес-интонацию;
- позу учителя;
- украшения без смысла;
- лишние вводные;
- фальшивую уверенность.

Не используй риторические конструкции противопоставления вроде «не X, а Y», «не первое, не второе, а третье» и похожие формулы. Если хочется написать сильное слово, объясни его фактом или замени.

## Стандартная структура карточки

Каждая карточка в базе знаний должна содержать все разделы ниже — в этом порядке:

```
## Карточка N. [Название ситуации или задачи]

### Ситуация
[Что происходит, какой запрос или момент — конкретно и узнаваемо]

### Почему важно
[Что теряется, если войти в ситуацию неправильно]

### Где обычно ломается
[Нумерованный список — **жирный заголовок + объяснение**]

### Как смотреть точнее
[Что нужно выяснить / как диагностировать ситуацию / правильный угол зрения]

### Что делать
[Пронумерованный список действий]

### [Опциональные разделы]
Вопросы клиенту / Что написать клиенту / Что сказать на созвоне /
Когда звать руководителя — по ситуации

### Артефакт
[Что должно появиться в результате: документ, письмо, список, решение]

### Критерий хорошего результата
[Как понять, что карточка отработана правильно]

### Главное правило
[Одна ёмкая формулировка — суть карточки]
```

## Frontmatter и таксономия

Для новых самостоятельных карточек и материалов добавляй YAML frontmatter. Для существующих тематических страниц в `знания/`, где карточки собраны внутри одного файла и используются HTML-генератором, сохраняй текущий формат с `_Обновлено: YYYY-MM-DD_` в начале.

Базовый frontmatter для нового самостоятельного материала:

```yaml
---
title: ""
type: "situation-card"
status: "draft"
section: ""
audience: ""
situation: ""
artifact: ""
visibility: "internal"
tags:
  -
related:
  - "[[]]"
created:
updated:
---
```

Типы материалов:
- `situation-card` — карточка рабочей ситуации;
- `principle` — принцип;
- `process` — процесс;
- `template` — шаблон;
- `case` — кейс;
- `checklist` — чеклист;
- `guide` — инструкция;
- `glossary` — словарь.

Разделы:
- `self`;
- `client`;
- `project`;
- `team`;
- `production`;
- `money`;
- `agency`.

Статусы:
- `seed` — есть мысль;
- `draft` — черновик;
- `review` — требуется проверка;
- `ready` — готово к публикации;
- `published` — опубликовано;
- `archive` — устарело.

Видимость:
- `internal` — для команды;
- `public` — можно публиковать;
- `private` — чувствительный материал.

Если материал содержит личные истории, конфиденциальные данные клиентов, деньги, конфликтные ситуации или имена людей, ставь `visibility: private` или `visibility: internal`.

## Правила работы

1. **Один файл — одна тема.** Не делай гигантских страниц.
2. **Всегда ставь дату последнего обновления** в начале файла: `_Обновлено: YYYY-MM-DD_`
3. **Используй `[[wikilinks]]`** для ссылок между страницами.
4. **Помечай неточности**: если информация неполная — пиши `> ⚠️ Требует уточнения:`
5. **Формат заголовков**: `# Название`, `## Раздел`, `### Подраздел`
6. **Карточки — по стандарту.** При создании или обновлении карточек следуй разделу «Стандартная структура карточки» выше.
7. **Связи обязательны.** У новой карточки должно быть минимум 3 связи: похожая ситуация, процесс или принцип, шаблон или кейс.
8. **Имена файлов — спокойные и понятные.** Например: `клиент-молчит.md`, `боюсь-написать-клиенту.md`, `вход-в-задачу.md`, `письмо-клиенту-о-задержке.md`, `постразбор-проекта.md`.

## Как обрабатывать входящие

Когда пользователь говорит "обработай входящие":
1. Найди все файлы в корне `входящие/` (кроме README.md и папки `обработано/`)
2. Для каждого файла: прочитай, определи темы, интегрируй в соответствующие страницы wiki
3. Если подходящей страницы нет — создай новую в нужном разделе
4. Обнови `_INDEX.md`
5. Перемести обработанный файл в `входящие/обработано/`

При разборе сырого материала:
1. Коротко пойми, о чём материал.
2. Найди рабочие ситуации.
3. Найди принципы.
4. Найди возможные кейсы.
5. Найди артефакты: письма, чеклисты, вопросы, решения.
6. Реши, какие файлы стоит создать или обновить.
7. Для новых материалов определи тип, раздел, статус и видимость.
8. Добавь связанные карточки.
9. Отметь вопросы, которые требуют уточнения.

Перед статусом `ready` проверь:
- есть ли понятная ситуация;
- есть ли польза для читателя;
- есть ли действие;
- есть ли артефакт;
- есть ли критерий результата;
- есть ли связанные материалы;
- не нарушена ли конфиденциальность;
- нет ли канцелярита;
- нет ли общих слов без опоры;
- текст можно читать без объяснений автора рядом.

## Как добавить новую страницу знаний (HTML автогенерация)

Когда нужно добавить новую страницу в HTML-версию сайта:

1. Создай `знания/новое-имя.md` по стандартной структуре карточки
2. Добавь строку в `_INDEX.md`
3. **Больше ничего делать не нужно** — `scripts/md-to-html.js` автоматически:
   - Сгенерирует `docs-src/новое-имя.html` (помечает `<!-- md-auto-generated -->`)
   - Добавит карточку в `docs-src/знания.html`
   - Зашифрует и опубликует на GitHub (следующий цикл LaunchAgent, max 30 мин)

**Важно:** файлы `docs-src/*.html` БЕЗ маркера `md-auto-generated` — ручная правка, автогенератор их не трогает.
Если нужно принудительно пересоздать авто-страницу — добавь маркер в начало HTML или удели файл.

## Публикация

На сайт можно передавать материалы со статусом:
- `ready`
- `published`

Не передавай на сайт материалы с видимостью `private`. Для `internal` нужна внутренняя версия сайта.

## Как генерировать документы

Когда просят создать документ (онбординг, инструкцию, презентацию):
1. Собери релевантные страницы wiki
2. Скомпилируй в `документы/название.md`
3. Адаптируй стиль под формат документа (не wiki-стиль)
