# Правила публикации

## Что считается готовым материалом

Материал можно публиковать, если у него:

- понятная ситуация или задача;
- ясная польза;
- есть следующий шаг;
- есть артефакт или критерий результата;
- проставлены теги;
- есть связанные материалы;
- указан статус;
- указана видимость.

## Какие материалы попадают на сайт

Публикуются материалы со статусом:

```yaml
status: ready
```

или

```yaml
status: published
```

## Что не публикуется

Не публиковать материалы со статусами:

```yaml
status: seed
status: draft
status: review
status: archive
```

Не публиковать материалы с видимостью:

```yaml
visibility: private
```

## Внутренняя и внешняя версия

### internal

Материал можно показывать команде.

### public

Материал можно показывать во внешней базе.

### private

Материал остаётся в Obsidian.

## Проверка конфиденциальности

Перед публикацией проверить:

- нет ли имён людей без необходимости;
- нет ли персональных конфликтов;
- нет ли финансовых деталей;
- нет ли данных клиента;
- нет ли внутренних условий договора;
- нет ли информации, которую нельзя показать подрядчикам или внешней аудитории.

Если сомневаешься, ставь:

```yaml
visibility: private
```

## Проверка стиля

Перед публикацией убрать:

- канцелярит;
- общие слова без доказательств;
- длинные вводные;
- декоративные фразы;
- повторение одной мысли;
- слишком уверенные выводы без опоры;
- риторические противопоставления.

## Проверка пользы

Материал готов, если после чтения понятно:

1. Что происходит.
2. Почему это важно.
3. Где риск.
4. Что сделать.
5. Как понять, что стало лучше.

## Обновление индекса

После публикации материала проверить, куда его добавить:

- в раздел;
- в блок «Что случилось?»;
- в блок «Шаблоны»;
- в связанные материалы;
- в маршрут обучения;
- в карту процессов.

## Версионирование

При существенном обновлении материала менять поле:

```yaml
updated:
```

Если материал больше не отражает текущий подход, поставить:

```yaml
status: archive
```

и добавить короткое объяснение в начало файла.
